# Development Flow 分步开发流程（一期）

> 本流程基于 `docs/plan.md` 细化，遵循“文档驱动开发”：先定规则，再做实现，每步可验收、可回滚、可联调。

## 0. 流程目标与原则

### 0.1 目标

在一期 12 周内，按“4 个核心服务 + 联调 + 上线”的节奏，完成 MVP 可用版本：

1. `enterprise-gateway-service`
2. `enterprise-knowledge-ai-service`
3. `enterprise-workbench-service`
   1. `enterprise-collaboration-service`


### 0.2 执行原则

1. 先文档后代码：若设计变化，先改文档再改代码。
2. 先骨架后功能：先打通登录、鉴权、网关和统一响应。
3. 先 P0 后 P1：先交付高频闭环，再增强 AI 和看板能力。
4. 每步必须可验收：接口可测、权限可测、日志可追踪。

## 2. 可执行开发流程（分步）

## Step 1：项目基线与工程骨架（阶段一-前半）

### 输入
1. `constitution.md`
2. `spec.md`
3. `plan.md`
4. `sdd.md`
5. `AGENTS.md`
6. `SKILL.md`

### 实施
1. 初始化前端工程与后端多服务工程结构。
2. 建立统一响应格式、全局异常处理、错误码规范。
3. 建立日志规范（请求日志、操作日志、traceId）。
4. 明确环境配置模板（dev/test/prod）。

### 输出
1. 可启动的前后端骨架。
2. 网关健康检查接口。
3. 统一返回结构在 demo 接口生效。

### 验收
1. 本地一键启动通过。
2. 任意异常都按统一格式返回。
3. 日志可看到 traceId 串联。

---

## Step 2：网关认证与 RBAC（阶段一-后半）

### 实施
1. 完成登录、退出、Token 签发与校验。
2. 完成用户、部门、角色、权限管理。
3. 网关路由转发 + 基础限流 + 黑白名单。
4. 接口默认鉴权（白名单除外）。

### 输出
1. `enterprise-gateway-service` 可作为统一入口。
2. 后台权限管理可用。

### 验收
1. 未登录访问受保护接口返回未授权。
2. 不同角色访问同一资源结果不同。
3. 管理操作有操作日志。

---

## Step 3：知识库最小闭环（阶段二-前半）

### 实施
1. 知识分类管理。
2. **逻辑知识库**管理（多 Milvus 集合、`kb_knowledge_base` + 文档 `kb_id`）。
3. 文档上传与元数据入库；**Apache Tika** 落盘解析；默认文档预处理；**异步分块**（`PENDING` → `RUNNING` → `SUCCESS`/`FAILED`）。
4. 文档权限策略（全员/部门/项目成员/指定人/管理员）。
5. 文档切片 CRUD、启用禁用、与 **Milvus** 向量写入（`id`/`content`/`metadata`/`embedding` Schema）；**分块任务日志**表。

### 输出
1. 分类、知识库、文档、切片相关 REST 接口（见 `docs/api.md` 第 3 节）。
2. 文档权限过滤与可见性校验。
3. Milvus 默认集合 + 按知识库集合；摄取与补偿接口。

### 验收
1. 上传并分块成功后可在文档列表检索到（`SUCCESS` 等可见状态）。
2. 无权限用户看不到文档数据。
3. 状态流转正确且有 **chunk 日志**与应用日志可追溯。

> 实现总览见 **`docs/step3-summary.md`**（含状态机、时序图、表结构与配置）。

---

## Step 4：Agent 智能检索 + MCP Server（阶段 2C）

### 实施
1. Agent 模块骨架搭建，新增 `kb_agent_session` / `kb_agent_message` 表。
2. LLM 客户端抽象 + Anthropic SDK 实现 + 配置（model、temperature 等）。
3. ToolRegistry + 4 个检索 Tool（search_documents / list_documents / get_document_detail / list_knowledge_bases）。
4. AgentLoop 循环实现 + POST `/api/kb/agent/chat` SSE 流式响应。
5. MCP Server 端点：`/mcp/sse`、`/mcp/tools/list`、`/mcp/messages`。
6. 会话管理接口：列表/历史/归档。
7. Tool 执行权限校验 + 参数校验 + 字段过滤。

### 输出
1. 自然语言知识检索能力。
2. 标准 MCP Server 暴露工具发现接口。
3. 会话持久化与历史查询。

### 验收
1. 自然语言提问能正确调用 Tool 并返回文档结果。
2. 无权限文档不会出现在 Tool 返回结果中。
3. SSE 流式输出正常。
4. MCP `/mcp/tools/list` 返回正确的 Tool Schema。

> 详细设计见 `docs/superpowers/specs/2026-05-10-agent-mcp-design.md`

---

## Step 4B：检索与 RAG 问答（阶段 2D + 2E）

### 实施
1. 文档全文索引、向量索引接入（Elasticsearch + Milvus Search）。
2. 混合检索编排（向量 + 全文 → rerank）。
3. RAG 问答流程：权限过滤检索 -> 组装上下文 -> 模型生成。
4. 返回答案时附来源引用。
5. 无可靠依据时返回”未找到明确依据”。
6. 落库 QA 会话、消息、反馈数据。

### 输出
1. 问答接口与多轮对话能力。
2. 问答反馈记录能力。

### 验收
1. 回答必须包含来源。
2. 无权限文档不会出现在来源或答案片段。
3. 证据不足时严格兜底，不编造。

---

## Step 5：协同业务核心闭环（阶段三）

### 实施
1. 会议室管理、会议创建、冲突检测（会议室+参会人）。
2. 会议修改/取消与通知触发。
3. 待办事项全流程（创建、更新、筛选、提醒、状态流转）。
4. 任务协同全流程（创建、分配、评论、状态流转、确认完成）。
5. 消息通知中心（列表、已读、未读计数、跳转）。
6. 基础数据看板统计接口。

### 输出
1. `enterprise-collaboration-service` 的核心业务接口。
2. 对应前端页面可联通。

### 验收
1. 冲突会议不能创建。
2. 任务分配会触发通知。
3. 待办/任务状态流转符合文档定义。

---

## Step 6：工作台聚合（阶段四）

### 实施
1. 聚合今日会议、今日待办、即将截止任务、未读通知。
2. 聚合常用知识与快捷入口。
3. 聚合接口统一返回结构并保证性能。

### 输出
1. `enterprise-workbench-service` 聚合接口。
2. 工作台首页前端页面。

### 验收
1. 首页关键卡片一次加载完成。
2. 聚合数据按当前用户权限过滤。

---

## Step 7：联调与权限打通（阶段五）

### 实施
1. 前端统一走网关访问所有后端服务。
2. 网关透传用户身份至下游。
3. 下游服务识别用户并执行数据权限过滤。
4. 统一跨服务异常与日志。

### 输出
1. 核心链路联调通过。
2. 权限链路全流程可验证。

### 验收
1. 从登录到各模块访问无断链。
2. 跨服务调用失败时返回统一错误结构。

---

## Step 8：测试、部署与上线准备（阶段六）

### 实施
1. 单元、接口、集成测试补齐。
2. 权限专项、会议冲突专项、AI 安全专项测试。
3. 部署脚本、监控告警、备份、回滚方案。
4. 上线检查清单与验收材料。

### 输出
1. 测试报告。
2. 部署手册。
3. 上线检查清单。

### 验收
1. 核心用例通过率达标。
2. 备份与回滚演练成功。

---

## 3. 分步开发建议（你可以这样推进）

为方便你“分步开发”，建议按 4 个里程碑推进：

1. **里程碑 A（已完成）**：Step 1 + Step 2  
   目标：网关、认证、RBAC 打通。
2. **里程碑 B（已完成）**：Step 3  
   目标：知识库最小闭环（上传、分块、Milvus 写入）。
3. **里程碑 C（当前，第 1-2 周）**：阶段 2B + Step 4  
   目标：一致性修复 + Agent 智能检索 + MCP Server 可用。
4. **里程碑 D（第 3-6 周）**：Step 4B + Step 5  
   目标：检索增强 + RAG 问答 + 协同业务主流程可用。
5. **里程碑 E（第 7-12 周）**：Step 6 + Step 7 + Step 8  
   目标：工作台聚合、联调、测试上线准备完成。

---

## 4. 每步开工前检查清单

1. 是否完成必读文档复核（6 份）？
2. 本步涉及接口/库表是否已在文档定义？
3. 权限与日志是否已纳入实现范围？
4. 是否定义了本步验收用例与回滚方案？

若以上任一项为“否”，先补文档/测试设计，再进入编码。
