# Plan 实施计划

## 1. 开发策略

一期建议采用“模块化单体架构”。这种方式可以降低早期开发和部署复杂度，同时在代码结构上保持清晰的模块边界，方便未来演进为微服务架构。

一期推荐结构：

1. frontend-web：前端项目。
2. backend-api：后端业务服务。
3. relational database：关系型数据库。
4. Redis：缓存。
5. file storage：文件存储。
6. search engine：全文检索。
7. vector database：向量数据库。
8. message queue：消息队列。

## 4. 关键风险

### 4.1 知识质量风险

如果公司内部文档过旧、格式混乱或缺少维护人，会导致搜索和智能问答效果下降。

应对措施：
1. 增加文档审核。
2. 设置文档负责人。
3. 支持版本管理。
4. 建立反馈机制。

### 4.2 权限风险

内部文档可能包含敏感信息。

应对措施：
1. 搜索、预览、下载和智能问答都要做权限校验。
2. 关键操作记录日志。
3. 管理员定期审计。

### 4.3 AI 幻觉风险

大模型可能生成没有依据的答案。

应对措施：
1. 要求答案必须基于检索来源。
2. 证据不足时返回无答案提示。
3. 增加用户反馈。
4. 严格控制提示词规则。

### 4.4 使用率风险

员工可能继续使用微信群、邮箱和本地文件，不愿迁移到新平台。

应对措施：
1. 工作台设计要足够简单。
2. 消息通知要及时。
3. 搜索和会议预约要明显比原方式方便。
4. 先选择一两个部门试点。
