## 1. 项目概述

### 1.1 项目定位

**企业智能工作平台（Enterprise Knowledge Workspace）** 是一套面向中大型组织的内部数字化工作系统。它将 **企业知识管理**、**AI 智能检索与问答**、**协同办公**（会议、待办、任务、审批、即时通讯、文档协作）以及 **管理后台** 整合在同一套 Web 应用中，形成「知识沉淀 → 智能消费 → 日常协作 → 管理治理」的闭环。

与传统单一功能系统不同，本平台的核心差异化在于：

- **知识库不是孤立模块**：上传的文档经过解析、分块、向量化后，可被 Agent 通过 RAG（检索增强生成）直接引用，员工在对话界面即可获取有依据的答案。
- **AI 不是外挂聊天框**：Agent 通过 MCP（Model Context Protocol）工具协议调用后端真实 API，执行文档搜索、知识库列表、会议创建等操作，并在 Tool 层做权限隔离。
- **工作台聚合跨域数据**：工作台服务通过 Feign 调用协同与知识库服务，将待办、会议、审批、未读消息、最近文档等聚合为单一 Overview，并用 Redis 缓存降低跨服务调用开销。

### 1.2 业务价值

| 维度 | 价值 |
|------|------|
| 知识复用 | 制度、技术文档、SOP 集中入库，减少重复问询 |
| 智能检索 | 自然语言提问 + 向量检索，降低查找成本 |
| 协同效率 | 任务看板、审批流、IM、协作文档在同一平台完成 |
| 管理可视 | 管理员可查看文档处理状态、用户统计、流水线任务 |
| 安全合规 | JWT + RBAC + 文档级权限 + 操作审计 |

### 1.3 系统边界

**已实现（MVP 可用）**：

- 网关认证与系统管理（用户/角色/权限/部门/操作日志）
- 知识库 CRUD、文档上传、异步分块、Milvus 向量写入、切片管理
- Agent SSE 对话、MCP Server、14+ Tool（含 RAG、会议、上传等）
- 协作文档（Quill + WebSocket OT）、即时通讯（WebSocket + RocketMQ）
- 会议、待办、任务、审批、公告、通讯录
- 工作台 Overview/Stats、Redis 缓存、收藏
- 意图树、关键词映射（智能路由到知识库）
- 管理后台：知识库 Hub、文档管理、流水线、Analytics 看板

**规划中或部分占位**：

- 统一消息通知中心（`/api/notifications` 设计存在，后端未实现）
- 全文检索独立模块（部分能力已通过 Agent RAG 落地）
- 部分管理页为前端 Mock（链路追踪、示例问题、系统设置）
- Gateway 对 `/api/kb/agent/**`、`/mcp/**` 路由需补全或文档化直连策略

┌─────────┐          ┌─────────────┐         ┌─────────────┐

### 2.2 分层架构

各业务服务内部遵循经典 Spring 分层：

| 层次 | 职责 | 典型包/类 |
|------|------|-----------|
| 表现层 | REST Controller、WebSocket Handler | `*Controller.java` |
| 业务层 | 领域逻辑、事务、事件 | `*ServiceImpl.java` |
| 持久层 | MyBatis-Plus Mapper / JPA Repository | `*Mapper.java`, `*Mapper.xml` |
| 集成层 | Feign、Milvus SDK、LLM Client | `MilvusVectorWriter`, `AgentLoop` |
| 横切 | 鉴权拦截、异常、日志、TraceId | `UserContextInterceptor`, `GlobalExceptionHandler` |

### 2.3 服务间通信模式

| 模式 | 场景 | 说明 |
|------|------|------|
| 同步 HTTP | 前端 → 后端、Workbench → Feign | REST + JSON，`Result<T>` 统一包装 |
| SSE 流式 | Agent 对话 | `POST /api/kb/agent/chat` 返回 `text/event-stream` |
| WebSocket | IM 聊天、协作文档 OT | 直连 collaboration :8090 |
| 异步事件 | 文档分块 | `@TransactionalEventListener` + `@Async` |
| 缓存失效 | 会议变更 → 工作台 | `WorkbenchCacheNotifier` HTTP 调用 |

### 2.4 三库分离

数据按业务域拆分为三个 MySQL Schema，避免单库耦合：

- **enterprise_gateway**：认证、RBAC、审计、Token 黑名单
- **enterprise_knowledge_ai**：知识库、文档、切片、Agent 会话、流水线
- **enterprise_collaboration**：协同业务、IM、意图树、关键词映射、工作台表（`wb_*`）

> 注意：主 SQL 脚本 `resouces/enterprise_knowledge_workspace.sql` 中部分意图表曾写在 knowledge 库块内，**运行时以 collaboration 的 migration 脚本为准**。

### 2.5 Gateway 真实 Filter 顺序（易错点）

生产经 `:8086` 的请求 **不是**「先 IP 限流再验 JWT」。正确顺序：

```text
① GatewayTraceIdFilter（WebFilter）
② Spring Security → JwtAuthenticationWebFilter（无 JWT 且非白名单 → 401，到不了后续 Filter）
③ IpAccessGlobalFilter（Order -100）
④ SimpleRateLimitGlobalFilter（Order -90）
⑤ … RoutePredicate / LoadBalancer …
⑥ IdentityPropagationGlobalFilter（Order 200）→ 注入 X-User-Id，移除 Authorization
⑦ NettyRoutingFilter → 转发下游
```

**含义**：未认证请求在步骤 ② 即被拦截；IP 黑白名单与限流只对 **已通过 JWT 或白名单** 的请求生效。详见 `docs/gateway-service-code-analysis.md` §5。

### 2.6 直连 vs 经 Gateway 路由表

| 流量类型 | 开发（Vite） | 生产建议 | 说明 |
|----------|-------------|----------|------|
| `/api/auth`、`/api/system` | → :8086 | → Gateway | 网关本地 Controller |
| `/api/kb/**`（含 agent） | → :8083 **直连** | Gateway 或 Nginx 反代 | Gateway 路由含 `/api/kb/**`，但 agent/mcp 易漏配 |
| `/mcp/**` | → :8083 直连 | 同 knowledge-ai | MCP SSE 长连接 |
| `/api/workbench/**` | → :8084 | → Gateway | |
| 其余 `/api/*` | → :8086 | → Gateway | meetings/todos/chat/docs… |
| `ws://…/ws/chat`、`/ws/docs` | → :8090 **直连** | Nginx Upgrade → :8090 | **不经 Gateway** |

## 4. 共享框架库 frameworks

`frameworks/` 不是可运行服务，而是被各业务模块依赖的 **Spring Boot Starter** 集合。设计原则是：**公共能力只写一次，禁止复制到业务模块**。

### 4.1 frameworks-common-spring-boot-starter

路径：`frameworks/common/src/main/java/com/zjl/common/`

**统一响应模型 `Result<T>`**：

```json
{
  "code": "200",
  "message": "success",
  "data": { ... },
  "traceId": "abc123..."
}
```

- 成功码为字符串 `"200"`（前端判断时需 `String(code) === '200'`）
- 所有响应携带 `traceId`，便于日志串联

**分页模型 `PageResult<T>`**：`current`, `size`, `total`, `records`

**错误码 `ErrorCode`（5 位）**：

| 码 | 枚举 | 含义 |
|----|------|------|
| 40000 | PARAM_INVALID | 参数校验失败 |
| 40100 | UNAUTHORIZED | 未登录或 Token 无效 |
| 40300 | FORBIDDEN | 无权限 |
| 40400 | NOT_FOUND | 资源不存在 |
| 50000 | SYSTEM_ERROR | 未捕获异常 |
| 50301 | VECTOR_WRITE_FAILED | Milvus 写入失败 |

**异常体系**：业务代码抛 `new BizException(ErrorCode.XXX)` 或带自定义 message；由 web starter 的 `GlobalExceptionHandler` 转为 `Result`。

**TraceId**：`TraceIdHolder` + MDC，Filter 注入。

### 4.2 frameworks-web-spring-boot-starter

| 组件 | 作用 |
|------|------|
| `GlobalExceptionHandler` | 捕获 BizException、ValidationException、兜底 Exception |
| `TraceIdFilter` | Servlet 链路透传 traceId |
| `ControllerLogAspect` | Controller 入参/耗时日志 |

Gateway 使用 WebFlux，仅依赖 common starter，自行实现响应式异常处理。

## 5. 微服务详解

### 5.1 enterprise-gateway-service（统一入口）

**职责**：对外唯一 HTTP 入口（生产环境）、JWT 签发与校验、RBAC、路由转发、限流、IP 黑白名单、系统管理 API。

**技术特点**：Spring Cloud Gateway + WebFlux + Spring Security Reactive + JPA。

**核心类**：

| 类 | 路径 | 说明 |
|----|------|------|
| `AuthController` | `com.zjl.web` | `POST /api/auth/login`, `POST /api/auth/logout` |
| `SystemAdminController` | `com.zjl.web` | 用户/角色/权限/部门/日志 CRUD，`@PreAuthorize("hasRole('ADMIN')")` |
| `JwtAuthenticationWebFilter` | `com.zjl.filter` | Bearer Token 解析 |
| `IdentityPropagationGlobalFilter` | `com.zjl.filter` | 向下游注入 `X-User-Id` 等 Header |
| `SimpleRateLimitGlobalFilter` | `com.zjl.filter` | 默认 120 req/60s/IP |
| `RbacUtil` | `com.zjl.security` | JWT claims → Spring Security Authentication |

**Gateway 路由表**（`application.yml`）：

| Route ID | 目标服务 | Path 谓词 |
|----------|----------|-----------|
| knowledge-ai | `lb://enterprise-knowledge-ai-service` | `/api/kb/**`, `/api/ai-qa/**` |
| collaboration | `lb://enterprise-collaboration-service` | `/api/meetings/**`, `/api/todos/**`, `/api/tasks/**`, `/api/chat/**`, `/api/docs/**`, `/api/approvals/**`, `/api/announcements/**`, `/api/intents/**`, `/api/keyword-mappings/**` |
| workbench | `lb://enterprise-workbench-service` | `/api/workbench/**` |

> **注意**：`/api/kb/agent/**` 与 `/mcp/**` 在部分环境未写入 Gateway 路由，开发期 Vite 直连 `:8083`；生产须在 Gateway 或 Nginx 显式配置。

**网关本地处理（不转发）**：`/api/auth/**`, `/api/system/**`, `/api/contacts/**`

**数据库表（enterprise_gateway）**：

- `sys_user`, `sys_dept`, `sys_role`, `sys_permission`
- `sys_user_role`, `sys_role_permission`
- `sys_op_log`（操作审计）
- `sys_token_blacklist`（登出 Token 哈希）

### 5.2 enterprise-knowledge-ai-service（知识库与 AI）

**职责**：知识库管理、文档摄取与分块、Milvus 向量、Agent 对话、MCP Server、流水线只读查询、管理统计。

**技术特点**：Spring MVC + MyBatis-Plus + `@EnableAsync` + Milvus SDK + Spring AI。

**包结构**：

```
com.zjl.knowledge
├── web/              # REST Controller
├── service/          # 业务接口与实现
├── entity/           # 数据库实体
├── mapper/           # MyBatis Mapper + XML
├── domain/           # DocumentStatus 等领域枚举
├── chunk/            # FIXED_SIZE / PARAGRAPH 分块策略
├── embedding/        # 百炼、DeepSeek Embedding
├── milvus/           # 向量读写、Collection 管理
├── event/            # DocumentChunkRequestedEvent 等
├── agent/            # AgentLoop、ToolRegistry、LLM Client
│   ├── tool/         # MCP Tool 实现
│   └── mcp/          # MCP Server Controller
└── config/           # MilvusProperties、AgentProperties
```

**主要 Controller 与 API**：

| Controller | 前缀 | 核心端点 |
|-----------|------|----------|
| `KbKnowledgeBaseController` | `/api/kb/bases` | 知识库 CRUD |
| `KbDocumentController` | `/api/kb` | 文档分页、上传、start-chunk、download、**document-stats** |
| `KbChunkController` | `/api/kb/documents/{docId}/chunks` | 切片 CRUD、批量启停 |
| `KbCategoryController` | `/api/kb/categories` | 分类树 |
| `KbAdminController` | `/api/kb/admin` | `GET /stats` 管理统计 |
| `PipelineController` | `/api/kb/pipelines` | 流水线列表/详情/任务 |
| `AgentController` | `/api/kb/agent` | SSE chat、会话 CRUD、upload |
| `McpServerController` | `/mcp` | SSE、tools/list、messages |

**关键实现类**：

- `KbDocumentServiceImpl` — 上传、权限过滤分页、触发分块
- `DocumentChunkingService` — Tika 解析、策略分块、向量写入、更新 chunk_count
- `DocumentChunkEventListener` — 事务提交后异步执行分块
- `MilvusVectorWriter` — Insert/Upsert/Delete，按 doc_id 或 chunk id
- `KbMilvusRoutingService` — 文档 kb_id → Milvus collection_name
- `AgentLoop` — LLM 多轮 Tool Call 循环
- `ToolRegistry` — Tool 注册、权限校验、执行分发

**下游身份解析**：`UserContextInterceptor` 读取 Gateway 注入的 Header：

- `X-User-Id`（必填）
- `X-Department-Id`, `X-Project-Id`, `X-Is-Admin`

## 7. 认证、鉴权与身份传递

### 7.1 双认证体系（重要）

当前存在 **两套 JWT 体系**，开发联调时需明确前端请求走哪条链路：

| 体系 | 服务 | 场景 |
|------|------|------|
| Gateway JWT | :8086 | `/api/system/*`、经 Gateway 转发的 API |
| Collaboration JWT | :8090 | 部分页面直连协同登录、WebSocket |

前端 Login 页调用 `POST /api/auth/login`（Vite 代理到 Gateway 或 Collaboration，视配置而定），返回 token 双写 `localStorage.token` 与 `user.token`。

### 7.2 Gateway 认证流程

1. 白名单路径（login、health、actuator）`permitAll`
2. 其余请求需 `Authorization: Bearer <JWT>`
3. 解析 JWT → 查 Token 黑名单 → 构建 Authentication
4. `IdentityPropagationGlobalFilter` 注入下游 Header，**移除 Authorization**（下游信任 Header）

**JWT 配置**：HS256，`ttl-seconds: 86400`（24 小时）

**登出**：Token SHA-256 写入 `sys_token_blacklist`

### 7.3 RBAC 模型

```
sys_user ←→ sys_user_role ←→ sys_role ←→ sys_role_permission ←→ sys_permission
```

- 角色 code `admin` 映射为 `ROLE_ADMIN`
- 管理接口 `@PreAuthorize("hasRole('ADMIN')")`
- 操作写入 `sys_op_log`

### 7.4 知识库文档权限

**文档级 `permission_type`**（`DocumentPermissionType` 枚举）：

| 值 | 含义 |
|----|------|
| ALL | 全员可见 |
| DEPARTMENT | 本部门可见 |
| PROJECT | 项目成员可见（配合 `kb_document_permission`） |
| USER | 指定用户可见 |
| ADMIN | 仅管理员 |

列表 SQL 在 `KbDocumentMapper.xml` 中按 `UserContext` 过滤；单文档与 Agent Tool 使用 `DocumentVisibilityService.canView(...)` **二次校验**。

**DocumentVisibilityService 决策（摘要）**：

| 条件 | 可见 |
|------|------|
| 用户 isAdmin | ✅ |
| doc.ownerId == userId | ✅ |
| permissionType=ALL | ✅ |
| DEPARTMENT 且 doc.departmentId == user.departmentId | ✅ |
| PROJECT / USER | ✅ 当 permission 表匹配 projectId/userId |
| permissionType=ADMIN（非 admin 用户） | ❌ |

RAG（`rag_qa` Tool）在 Milvus 检索后必须再过滤，防止向量库返回无权限文档 chunk。

## 8. 知识库与文档处理子系统

### 8.1 核心概念

| 概念 | 说明 |
|------|------|
| 逻辑知识库 `kb_knowledge_base` | 业务分组，绑定独立 Milvus `collection_name` 与可选 `embedding_model` |
| 文档 `kb_document` | 上传文件的元数据、状态、权限、chunk_count |
| 切片 `kb_document_chunk` | 分块正文、索引、向量 ID、启用状态 |
| 分块日志 `kb_document_chunk_log` | 每次分块任务耗时与结果 |
| 流水线 `kb_pipeline` | 与知识库 1:1，描述入库阶段（上传→解析→分块→向量→回写） |

### 8.2 文档状态机（DocumentStatus）

**主路径（已实现）**：

```
[*] → PENDING      （upload 成功）
PENDING → RUNNING  （start-chunk，乐观 CAS）
RUNNING → SUCCESS  （分块+向量+DB 全部成功）
RUNNING → FAILED   （任一步骤异常）
FAILED → RUNNING   （修正后重新 start-chunk）
SUCCESS → RUNNING  （重新分块，会清空旧 chunk）
```

**预留枚举**（较少使用）：DRAFT, PARSING, REVIEWING, PUBLISHED, REJECTED, OFFLINE

### 8.3 分块策略

| 策略 | 枚举 | 说明 |
|------|------|------|
| 固定长度 | FIXED_SIZE | 按字符/token 窗口切分 |
| 段落 | PARAGRAPH | 按段落边界切分 |

配置存于文档 `chunk_strategy` + `chunk_config` JSON。

### 8.4 异步分块流程

1. `POST /api/kb/documents/upload` — multipart，落盘，INSERT status=PENDING
2. `POST /api/kb/documents/{id}/start-chunk` — CAS 更新 RUNNING，发布 `DocumentChunkRequestedEvent`
3. `DocumentChunkEventListener`（AFTER_COMMIT + Async）调用 `DocumentChunkingService.executeChunk`
4. Tika 提取全文 → 策略分块 →（可选）Embedding → 事务内写 MySQL chunk + 更新 document.chunk_count + Milvus Insert
5. 成功 → SUCCESS；失败 → FAILED + 日志

补偿接口：`POST /documents/{id}/execute-chunk`（同步执行，运维/调试用）

### 8.5 Milvus 集成

**Schema（每 collection）**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VarChar PK | = chunk.id 字符串 |
| content | VarChar(65535) | 正文，超长截断 |
| metadata | JSON | doc_id, chunk_index, collection_name |
| embedding | FloatVector | 维度默认 1024，COSINE |

**路由**：文档 `kb_id` → `kb_knowledge_base.collection_name`；无 kb_id 用默认 `app.milvus.collection`。

**开关**：

- 全局：`app.knowledge.vector-write-enabled`
- 模型：`app.knowledge.embedding-model`（空则跳过向量化）
- 知识库级：`embedding_model` 字段优先

**操作**：Insert / Upsert / 按 doc_id 删除 / 按 chunk id 删除；文档禁用时同步向量状态。

### 8.6 存储

- 本地：`app.kb.storage.upload-dir`（默认 `./data/kb-uploads`）
- 生产：`app.kb.storage.type=s3`，阿里云 OSS 兼容

### 8.7 管理统计 API

| 端点 | 返回 |
|------|------|
| `GET /api/kb/document-stats` | totalDocs、各状态文档数、docSuccessRate、**totalChunks** |
| `GET /api/kb/admin/stats` | 分块日志成功率、Agent 消息统计、文档统计等 |

`totalChunks` 统计 `kb_document_chunk` 表行数，为 KnowledgeHub「累计切片」卡片的数据源。

## 14. 核心业务流程

### 14.1 用户登录

```
用户 → Login.vue → POST /api/auth/login
     → 服务端 BCrypt 验密 → 签发 JWT
     → localStorage 存储 user + token
     → 跳转 Dashboard
后续请求 → getAuthHeaders() → Bearer + X-* Headers
401/40100 → forceLogout → 登录页 + 提示消息
```

### 14.2 知识入库闭环

```
管理员 → admin/Documents 上传文件
      → POST /documents/upload (PENDING)
      → POST /documents/{id}/start-chunk
      → 异步：Tika → 分块 → Embedding → MySQL + Milvus
      → SUCCESS
员工 → Chat 提问 → Agent rag_qa Tool → Milvus 检索 → LLM 生成答案
```

### 14.3 Agent 问答

```
用户输入 → AgentController.chat (SSE)
        → AgentLoop: LLM + tools schema
        → tool_call → ToolRegistry → 知识库/会议 API
        → tool_result 回填 LLM
        → 流式 delta 输出
        → 持久化 kb_agent_message
```

### 14.4 审批流转（请假）

```
员工提交 → status=pending
主管审批 → manager_approved 或 rejected
（请假）→ approved
工作台 Overview 展示 pendingApprovals 计数
```

### 14.5 IM 消息

```
客户端 WS 连接 /ws/chat?token=
发送消息 → REST POST 或 WS 帧
服务端 → 持久化 im_message → RocketMQ → 推送给在线成员
未读数 → GET /api/chat/unread-count（MainLayout 60s 轮询）
```

## 16. 开发规范与质量约束

规范权威文件：**`docs/AGENTS.md`**、**`docs/constitution.md`**

### 16.1 代码规范

- **注释**：仅 Javadoc `/** */`，禁止 `//` 行注释（constitution 要求）
- **响应**：Controller 返回 `Results.success(data)`，禁止裸对象
- **异常**：抛 `BizException`，禁止吞异常
- **删除**：关键业务逻辑删除，用 `deleted` 字段
- **共享代码**：只用 frameworks starter，禁止复制

### 16.2 MyBatis-Plus 约定

- Entity 在 `entity/`，Mapper 继承 `BaseMapper<T>`
- Service 继承 `ServiceImpl<Mapper, Entity>`
- 复杂 SQL 写 `resources/mapper/*.xml`
- 分页：`Page<T>` + `selectPage`
- ID：雪花 `IdWorker.getId()`

### 16.3 校验

- Request DTO 使用 Jakarta Validation
- Controller 参数 `@Valid`
- 失败 → `PARAM_INVALID`

### 16.4 状态枚举

- 文档：`DocumentStatus`（PENDING/RUNNING/SUCCESS/FAILED 为主路径）
- 勿自创字符串状态，与 `docs/AGENTS.md` 保持一致

### 16.5 安全

- 新公开接口必须加入 Gateway `app.security.whitelist.paths`
- Agent Tool 必须校验文档可见性
- 生产必须更换 JWT secret

| 管理后台 | ★★★☆☆ | 部分页面 Mock |
| 消息通知中心 | ★☆☆☆☆ | 仅公告，无统一 Notification API |
| 全文检索 | ★★☆☆☆ | 标题搜索 + RAG，无 ES |

### 17.2 已知不一致（全平台）

1. **端口文档**：部分旧文档写 knowledge 8081，实际 **8083**
2. **双 JWT + 双 sys_user**：Gateway 与 Collaboration 各有一套用户表与 JWT secret；WebSocket 用 Gateway token 需 **secret 一致** 且 `JwtClaimsSupport` 读 sub
3. **Gateway Filter 顺序**：常见误解为「先限流后 JWT」，实际 **先 Security 后 GlobalFilter**
4. **Gateway 路由缺口**：生产需确认 `/api/kb/agent/**`、`/mcp/**` 可达 knowledge-ai
5. **意图表库归属**：master SQL vs migration 不一致，以 **collaboration** migration 为准
6. **Workbench intentCount**：Feign 调不存在的 `/api/intents` 根路径 → 恒 0
7. **路由守卫**：前端仅检查 `localStorage.user`，不验证 JWT 过期
8. **WebSocket**：硬编码 `:8090`，生产需 Nginx Upgrade
9. **IM 文件下载**：前端调 `GET /api/chat/files/{ossKey}`，后端未实现
10. **Task 列表**：Collaboration `GET /api/tasks` 全库不过滤 assignee
11. **PIPELINE 分块模式**：knowledge 枚举存在，主路径未实现
12. **Flyway**：collaboration 默认关闭，迁移需手工执行

## 19. 跨服务 Gotchas 汇总

按服务归纳「读源码才容易发现」的行为，详细展开见各 `*-service-code-analysis.md` 附录 Gotchas。

### 19.1 Gateway

| 项 | 说明 |
|----|------|
| JWT 先于 IP/限流 | 无 Token 请求不会进入 IpAccessGlobalFilter |
| 下游无 Authorization | IdentityPropagation 只传 X-* Header |
| 黑名单持久化 | DB `sys_token_blacklist`；Collaboration logout 为内存 Set |
| WebFlux + JPA | 网关用 JPA，注意阻塞调用勿占事件循环 |

### 19.2 Knowledge-AI

| 项 | 说明 |
|----|------|
| 上传≠可检索 | upload 后必须 start-chunk → SUCCESS |
| shouldEmbed | 全局开关 + embedding 模型非空才写 Milvus |
| 删除顺序 | 先删 Milvus 向量再删 MySQL |
| Agent 线程 | new Thread 非池化 |
| RagQa | 向量检索后 DocumentVisibilityService 过滤 |
| Milvus fail-on-init | 默认 false，向量失败可能静默 |

### 19.3 Collaboration

| 项 | 说明 |
|----|------|
| JwtAuthFilter | 有 X-User-Id 即放行；Bearer 只读 userId claim 不读 sub |
| IM 双推 | Service + MQ Consumer 各 push 一次 |
| Doc OT 锁 | 内存 ReentrantLock，多实例不安全 |
| 会议 /my | attendees 字符串 contains 匹配 |
| Zoom | 可选，失败降级本地会议 |

### 19.4 Workbench

| 项 | 说明 |
|----|------|
| wb_stats 缓存 | 全局 key，非 per-user |
| getTodos 两次 | overview 重复 Feign |
| intentCount | 下游 API 不存在 |

## 结语

企业智能工作平台是一个 **知识驱动、AI 增强、协同一体** 的内部数字化系统。其后端以四个 Spring Boot 微服务分工明确：网关守入口、知识库承载 AI 与向量、协同覆盖日常办公、工作台聚合跨域视图；前端以 Vue 3 统一体验，飞书风格主应用 + 深色管理后台。

理解系统的关键路径有五条：

1. **文档上传 → 分块 → Milvus → RAG 问答** 的知识闭环  
2. **JWT → Gateway Filter → X-* Header → 文档权限 SQL + DocumentVisibilityService** 的安全链路  
3. **Feign 聚合 → Redis 缓存 → WorkbenchCacheNotifier 主动失效** 的工作台性能链路  
4. **Agent SSE → ToolRegistry → CollaborationClient / VectorSyncService** 的 AI 行动链路  
5. **WebSocket 直连 :8090** 与 **Gateway REST :8086** 并存的混合入口链路  

新成员建议按「登录 → Dashboard → Chat 提问 → admin/Documents 上传 → KnowledgeHub 看状态 → Chats/Documents 体验 WS」顺序走通一遍，再按需深入 §21 索引中的四份服务分析文档。

本文档 v2.0 与四份 `*-service-code-analysis.md` 互补：本文偏 **全景与交叉关系**，服务文档偏 **单模块源码级细节**。若发现与实现不一致，以各服务源码及 `application.yml` 为准。
